You have two separate things happening now:

1. **The Xorg forced session is broken and should be removed.** Your normal GNOME session works; the fake `GNOME Xorg (forced)` session loops back to GDM. Current Arch GNOME packaging is Wayland-first, and the current `gnome-session` file listing shows the Wayland session entry, not a normal stock `/usr/share/xsessions/gnome-xorg.desktop` path we can rely on. The login manager normally starts `gnome-session`, so a fake Xorg desktop entry can easily produce exactly the login-loop behaviour you saw. ([Arch Manual Pages][1])

2. **We should stop depending on Xorg tools for the Super key.** On Wayland, `xcape`/`xrandr` are the wrong foundation. For bare `Super`, we will use `keyd`, because it remaps keys at a system input level and works across Wayland, Xorg, and TTY sessions. ([Arch Manual Pages][2])

Dash-to-Panel remains the correct dock base because it combines launchers, running apps, and system tray/status items into one panel, which is exactly why your battery/network/sound/clock are appearing inside the dock area. ([GNOME Extensions][3])

Run these **four v9 scripts** from the working normal **GNOME** session, not from `GNOME Classic`, and not from the broken `GNOME Xorg (forced)` option.

pass() { echo "PASS: $1" | tee -a "$OUT"; }
warn() { echo "WARN: $1" | tee -a "$OUT"; }
info() { echo "INFO: $1" | tee -a "$OUT"; }

getsetting() {
    gsettings get "$1" "$2" 2>/dev/null || echo "unavailable"
}

check_command() {
    if command -v "$1" >/dev/null 2>&1; then
        pass "command exists: $1 -> $(command -v "$1")"
    else
        warn "missing command: $1"
    fi
}

check_file() {
    if [ -e "$1" ]; then
        pass "file exists: $1"
    else
        warn "file missing: $1"
    fi
}

check_pkg() {
    if pacman -Q "$1" >/dev/null 2>&1; then
        pass "package installed: $1"
    else
        warn "package missing: $1"
    fi
}

    check_command "$cmd"

    pass "keyd service is active"

    warn "keyd service is not active"

info "Custom keybindings: $(getsetting org.gnome.settings-daemon.plugins.media-keys custom-keybindings)"

    pass "F13 custom keybinding slot for tapped Super exists"
else
    warn "F13 custom keybinding slot missing"

echo | tee -a "$OUT"
echo "Dock extensions" | tee -a "$OUT"
gnome-extensions list --enabled | tee -a "$OUT" || true

    warn "ArcMenu still enabled"
else

info "panel size: $(getsetting "$DTP" panel-size)"

echo | tee -a "$OUT"

check_file "$HOME/.config/wofi/style.css"

if grep -q 'IB_MACOS_DOCK_BEGIN' "$DTP_DIR/stylesheet.css" 2>/dev/null; then
    pass "Dash-to-Panel CSS patch is present"
else
    warn "Dash-to-Panel CSS patch is missing"

echo | tee -a "$OUT"

if systemctl is-active --quiet power-profiles-daemon.service; then
    pass "power-profiles-daemon active"

    warn "power-profiles-daemon not active"

echo | tee -a "$OUT"
echo "Report saved to: $OUT" | tee -a "$OUT"

One important limitation: the **true dynamic-width macOS dock** you described is not natively exposed by Dash-to-Panel. Dash-to-Panel is a GNOME panel/taskbar extension, not a pure CSS dock. We can make it 75% max-width-style, glassy, bordered, centred, and integrated with battery/network/sound/clock, but true “grow only to content width and then stop at 75%” will require a custom GNOME Shell extension or a separate AGS/EWW dock layer. For now, this v9 set keeps the stable integrated GNOME panel and fixes the broken session/key/menu/dock issues first.
